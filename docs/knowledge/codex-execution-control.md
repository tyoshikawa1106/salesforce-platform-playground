# Codex の認証操作と実行制御

Codex に継続して守らせる指示と、コマンドやツールの実行を機械的に制御する設定は役割が異なります。認証に関する操作境界を設計するときは、既存の認証状態を利用する通常の開発操作と、新しい認証状態を作成または変更する操作を分けて考えます。

## 認証操作の境界

`git`、`gh`、`sf` などが、開発環境ですでに利用可能な認証状態を使って処理すること自体は、新規ログインや再認証ではありません。通常の Git 操作や `git log` に含まれるコミットメタデータも、認証情報管理機能へのアクセスとは別のものです。

一方、ブラウザや認証情報管理アプリを起動して新規ログイン、追加認証、再認証を開始することや、OS の認証情報管理機能へ直接アクセスすること、認証設定を変更することは、認証状態を作成または変更する操作です。これらは通常の CLI 利用と分け、必要な場合に明示的な依頼を受けて実施します。

このリポジトリで実際に守る境界は、ルートの `AGENTS.md` を正とします。この文書は設定方式を振り返るためのナレッジであり、コマンド単位の許可リストではありません。

## 制御方法の使い分け

| 方法                | 主な役割                                             | 認証操作の制御に使う場合の注意点                                                                |
| ------------------- | ---------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| `AGENTS.md`         | リポジトリで継続して守る判断基準や作業手順           | 通常操作と新規認証を文脈に応じて区別できるが、ツール実行を機械的に遮断する仕組みではない        |
| `config.toml`       | sandbox、承認方針、承認者などの既定値                | 実行境界全体を設定するもので、特定の認証操作だけを意味で判別する設定ではない                    |
| Rules               | sandbox 外で実行するコマンド接頭辞の許可・確認・禁止 | CLI コマンドには使えるが、Rules は実験的機能であり、ブラウザや GUI を含む認証操作全体は扱えない |
| Hooks               | ツール実行前後などのイベントで独自スクリプトを実行   | `PreToolUse` などで検査できるが、対象イベントと実装した判定の範囲に限られる                     |
| `requirements.toml` | 組織管理者がローカル実行ポリシーを強制               | 個人やリポジトリの通常設定ではなく、管理対象クライアントへ配布する組織向けの仕組み              |

## `config.toml` と承認レビュー

個人の既定値は `~/.codex/config.toml`、信頼済みリポジトリ固有の設定は `.codex/config.toml` に置けます。代表的な実行制御は次の組み合わせです。

```toml
sandbox_mode = "workspace-write"
approval_policy = "on-request"
approvals_reviewer = "user"
```

`approvals_reviewer = "user"` は、すでに承認が必要と判定された操作をユーザーへ提示します。sandbox 内で許可されている操作へ新しい確認を追加する設定ではなく、認証操作だけを選んで止める設定でもありません。

`approvals_reviewer = "auto_review"` は、対象となる承認要求を別のレビュアーエージェントへ送ります。これは承認者を切り替える設定であり、sandbox の書き込み範囲やネットワーク境界を広げるものではありません。

## Rules と Hooks

Rules は、アクティブな設定レイヤーに隣接する `rules/` 配下の `.rules` ファイルで、コマンド接頭辞ごとに `allow`、`prompt`、`forbidden` を指定します。通常の開発コマンドまで広く止めないよう、対象を具体的なコマンド接頭辞へ限定する必要があります。

Hooks は、`hooks.json` または `config.toml` の `[hooks]` で定義します。`PreToolUse` はツール実行前の検査に利用できますが、複数の一致する Hooks は並行して開始されるため、1つの Hook が別の Hook の開始を防ぐ仕組みではありません。プロジェクト固有の Hooks は、プロジェクトの `.codex` レイヤーが信頼されている場合だけ読み込まれます。

Rules や Hooks を追加する場合は、次を先に確認します。

- 守りたい境界が、コマンド文字列だけで正確に判定できるか。
- 既存認証を利用する通常操作まで止めないか。
- CLI 以外のブラウザ、アプリ、MCP ツールも対象にする必要があるか。
- 誤検知時に安全な通常作業が継続できるか。
- 設定を個人、リポジトリ、組織管理のどの範囲へ適用するか。

## このリポジトリでの判断

現在は、認証操作の境界を `AGENTS.md` に置き、追加の `config.toml`、Rules、Hooks は導入しません。既存認証を利用する通常操作を維持しながら、文脈上の新規認証や再認証だけを区別するには、現時点ではリポジトリルールが最も狭い変更範囲だからです。

同じ種類の認証操作が再発し、文章による指示だけでは境界を維持できないことが確認された場合に、対象ツールと誤検知範囲を特定して Rules または Hooks を再検討します。
