# エージェントルールと GitHub Actions の使い分け

開発運用を自動化するときは、エージェントに任せるのか、GitHub Actions に任せるのかを先に決めます。

## 基本の考え方

エージェントルールは、Codex などの作業者が守る運用手順です。作業中に文脈を読んで判断できる一方で、エージェントが関わらない操作には効きません。

GitHub Actions は、リポジトリ上のイベントで必ず動く仕組みです。人がブラウザで操作した場合にも効きますが、権限、失敗時の扱い、例外処理を明確に設計する必要があります。

## エージェントルールに向いているもの

- 作業者が文脈を読んで判断する必要がある。
- 失敗したらユーザーに確認して止まる方が安全。
- Codex が作る Issue、PR、コミットだけに効けばよい。
- ルール違反があっても、即時に CI を失敗させるほどではない。
- まだ運用を試している段階で、軽く始めたい。

例:

- PR 作成時に対応 Issue のラベルを確認し、PR ラベルを揃える。
- PR マージ後に条件を確認して、ローカル作業ブランチを整理する。
- ドキュメント変更だけなら `git diff --check` を基本の検証にする。

## GitHub Actions に向いているもの

- 誰が操作しても必ず同じルールを適用したい。
- ブラウザ操作、外部 contributor、手動 PR にも効かせたい。
- ルール違反を CI failure や warning として明確に出したい。
- 作業者の注意に依存させたくない。
- 定型処理で、例外が少ない。

例:

- PR 本文の `Closes #123` から Issue ラベルを読み、PR に自動付与する。
- PR title の形式を検証する。
- required check として lint、test、metadata validation を実行する。

## 判断基準

| 観点           | エージェントルール       | GitHub Actions                  |
| -------------- | ------------------------ | ------------------------------- |
| 適用範囲       | エージェントが関わる作業 | GitHub 上の対象イベント         |
| 強制力         | 運用ルールとして守る     | CI や自動処理として強制できる   |
| 例外対応       | 会話で確認しやすい       | workflow 側に条件分岐が必要     |
| 導入コスト     | 低い                     | 中程度                          |
| 権限設計       | 作業者の権限に依存       | `GITHUB_TOKEN` などの設計が必要 |
| 失敗時の扱い   | ユーザー確認に戻しやすい | failure、warning、skip を設計   |
| 向いている段階 | 運用を固める前、試行段階 | ルールが固まり、強制したい段階  |

## 段階的な進め方

最初はエージェントルールで始め、運用が安定してから GitHub Actions に移すのが扱いやすいです。

1. エージェントルールで作業手順を明文化する。
2. 数回運用して、例外や停止条件を確認する。
3. 例外が少なく、強制したいルールだけ GitHub Actions 化する。
4. Actions 化した後も、エージェントルールには「結果を確認する」役割を残す。

## 注意点

GitHub Actions は強制力があるため、曖昧な運用をそのまま実装すると、不要な CI failure や権限エラーを増やします。

エージェントルールは軽く始められますが、人手やブラウザ操作には効きません。チーム全体で必ず守りたいルールは、最終的に GitHub Actions や branch protection に寄せます。
