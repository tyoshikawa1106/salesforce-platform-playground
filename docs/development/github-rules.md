# GitHub 運用ルール

GitHub Flowで変更を管理するときの共通ルールです。Issue、Project、Milestone、ラベル、Release、CIなどの利用有無はリポジトリごとに異なるため、この文書では前提にしません。

対象リポジトリに固有ルールがある場合は、共通ルールを基本とし、固有ルールを追加適用します。

## 基本ルール

- GitHub作業の実行可否とユーザー確認の境界は、対象リポジトリの`AGENTS.md`に従う。
- 変更前に現在のブランチ、作業ツリー、リモート、既存PRを確認する。
- デフォルトブランチへ直接コミットせず、作業ブランチからPull Requestを作成する。
- ブランチ名には作業内容が分かる短い要約を入れる。
- フックが依存不足で失敗した場合、勝手に依存を導入せず確認する。
- Issue、PRテンプレート、ラベル、担当者、Project、Milestone、Release、CI、ブランチ保護は、対象リポジトリの固有ルールと現在のGitHub設定を確認する。

## GitHub Flowの実行順序

1. 現在のリポジトリの状態と作業対象を確認する。
2. デフォルトブランチから作業ブランチを作成する。
3. 変更、ローカル検証、コミットを行う。
4. ローカルコミット後に一度停止する。
5. ユーザーの明示依頼がある場合だけ、プッシュ、PR作成、CI確認へ進む。
6. 必須チェックと未解決のレビュー指摘を確認する。
7. ユーザーの明示依頼がある場合だけPRをマージする。
8. マージ後にデフォルトブランチを同期し、条件を満たす場合は作業ブランチを整理する。

対象リポジトリでIssueや外部サービスとの連携が必要な場合は、固有ルールに定義された確認をこの順序へ追加します。

## GitHub CLIの認証判定

GitHub CLIの認証状態は、単一コマンドの失敗だけで認証切れと断定しません。

1. `gh auth status`で保存済み認証の状態を確認する。
2. 失敗または警告がある場合は、`gh api user`で認証済みAPI呼び出しが成功するか確認する。
3. サンドボックス、keyring、ネットワークなど実行環境の制約が疑われる場合は、許可された通常環境で同じ確認を再実行する。
4. `401`または`Bad credentials`など、資格情報が無効であることを示す応答が再現した場合だけ認証切れと判断する。

エラーは次のように区別します。

- `error connecting`、名前解決失敗、タイムアウトはネットワーク障害として扱い、認証切れの根拠にしない。
- keyringや資格情報ストアへのアクセス失敗は実行環境の問題として扱い、保存済み認証が無効とは判断しない。
- `gh api user`が成功した場合は認証済みと判断し、`gh auth status`単独の警告から再認証しない。
- 認証状態を確定できない場合は「認証状態を確認できない」と報告し、「認証切れ」と報告しない。

`gh auth login`の実行やブラウザの認証画面表示は、認証切れを上記手順で確認し、ユーザーから再認証の明示確認を得た後だけ行います。

## PR作成とマージ

- PR本文は変更内容、変更理由、影響範囲、確認結果をレビューできる内容にする。
- 対象リポジトリにPRテンプレートがある場合は、その構成に従う。
- 必須CI、レビュー、外部検証など、対象リポジトリのマージ条件を満たしてからマージする。
- マージ方法は対象リポジトリの設定または固有ルールに従う。

## PR マージ後の作業ブランチ整理

PR マージ後のデフォルトブランチ同期、ローカルブランチ削除、リモートブランチ削除の実行条件は、対象リポジトリの固有ルールに従います。固有ルールがない場合は自動実行せず、ユーザーに確認します。

## コミット本文

実質的な変更を含むコミットは、コミット件名だけで終わらせず本文を付けます。本文には、後から変更理由と確認結果を判断できる情報だけを簡潔に残します。

- コミット件名から分からない背景、問題、判断理由を記載する。
- 複数の重要な変更や判断境界がある場合は、主な変更内容を記載する。
- 最後の段落を`検証:`で始め、実行した確認コマンドと結果を記載する。必要な検証を実行していない場合は理由も記載する。
- コミット件名を言い換えただけの本文や、変更ファイルの機械的な列挙だけで終わらせない。
- 改行は実際の改行として入力し、本文に`\n`などのエスケープ表現を残さない。
- 実ユーザー名、メールアドレス、環境固有のユーザー名やID、秘密情報を記載しない。

次のように判断を伴わない機械的・軽微なコミットでは、本文を省略してよいです。

- 誤字修正のみ。
- フォーマットのみ。
- 同じ生成手順による生成物更新のみ。
