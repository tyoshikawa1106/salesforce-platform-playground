# 機能仕様書ルール

AI エージェントが現行実装の機能仕様書を追加、更新、分割するときは、このルールに従います。

## このルールの使い方

次の順序で判断します。

1. 対象が独自開発した機能か確認する。
2. 主仕様書を作る機能入口と、そこに含める実装・設定を決める。
3. 必須構成と、該当する条件付き記載事項に従って現行実装を記載する。
4. 要求や外部契約との比較結果を「既知の差異・確認事項」に記載する。
5. 安全・機密情報、関連リンク、整形を確認する。

## 文書の位置付けと正

- `docs/specifications/` は、実装、メタデータ、テストから確認できる現行実装の仕様と仕組みを記録する。
- 現行実装の説明についてはリポジトリ内の実装を正とする。
- 承認済みの要求、業務ルール、外部契約については、それぞれの管理元を正とし、現行実装仕様で上書きしない。
- 要求と実装が異なる場合は、どちらかを推測で修正せず、「既知の差異・確認事項」に事実、影響、確認先を記載する。不具合か許容差異かは承認者が判断する。
- 実装予定、承認前の要求、検討中の内容、判断過程は現行実装仕様に混在させず、Issue または `docs/discussions/` で管理する。
- 実装から確認できない内容を推測で仕様として固定しない。不明点は「既知の差異・確認事項」または制約として明記する。
- Salesforce や開発技術に関する汎用的な説明は `docs/knowledge/` に置く。
- 機能名、ページタイトル、リンク表示名は、日本語だけで役割を把握できる表現にする。
- API 名、クラス名、実際の画面表示、メタデータの状態値は、コード表記または日本語訳との併記で残す。

## 仕様書の対象範囲

仕様書の対象は、このリポジトリで独自実装した自動化、効率化、画面、連携などの開発機能です。Salesforce の設定全体を仕様書で網羅することは目的にしません。

### 対象にするもの

- Trigger、Batch、Scheduler、Queueable、Invocable、REST など、独立した振る舞いを提供する Apex
- Flow、LWC、Aura、Visualforce、独自 API、外部連携、イベント処理
- 独立した振る舞いと変更影響を持つ独自共通基盤

Controller、Service、Selector、共通モジュールなど、単独では機能入口にならない実装は、それを利用する主仕様書の「対象実装・メタデータ」に記載します。

### 独立した仕様書を作らないもの

次の Salesforce 設定は、それ自体の独立した仕様書や網羅的な一覧を作成しません。調査が必要なときはリポジトリ内のメタデータまたは対象組織の設定を確認します。

- オブジェクト、項目、レコードタイプ、選択リスト、数式、入力規則、重複ルール
- 権限セット、権限セットグループ、共有設定、共有ルール、Restriction Rule
- Custom Metadata、Custom Settings、Named Credential、External Credential
- Approval Process、Assignment Rule、Escalation Rule、Email Alert
- レポート、ダッシュボードなどの設定単体

独自開発機能がこれらの設定に依存する場合は、機能の理解、実行条件、権限、変更影響の把握に必要な範囲だけ、その機能仕様書へ完全な API 名と役割を記載します。設定の全項目や管理画面の内容は転記しません。

## 安全・機密情報

機能仕様書には、機能の理解に必要な契約、方式、設定名だけを記載し、秘密情報や実データを記載しません。

- token、password、client secret、秘密鍵、証明書、認証ヘッダー、セッション情報などの実値を記載しない。
- 実ユーザー名、メールアドレス、顧客情報、個人情報、org ID、レコード ID、tenant ID など、個人または環境を特定できる実値を記載しない。
- Named Credential、External Credential、接続アプリ、認証設定は、機能の理解に必要な API 名、役割、認証方式までを記載する。credential、principal、org 固有の接続先などの実値は記載しない。
- 外部 API は必要な公開契約を記載できるが、秘密を含む URL、クエリ文字列、ヘッダー、内部環境固有のホスト名は記載しない。
- 入出力例やエラー例には、実在データと誤認されない架空値または伏せ字を使用する。
- `.env`、認証ファイル、秘密鍵など、秘密情報を含み得るファイルを仕様書作成のために読まない。確認できない設定値は推測せず、確認事項として残す。
- 調査中に秘密情報や実データを発見しても仕様書へ転記しない。必要な場合は値を伏せ、機能への影響と安全な確認先だけを記載する。

## 仕様書の単位

主仕様書は、利用者、外部システム、スケジュール、イベント、データ変更などから処理を開始できる機能入口を単位とし、専用ディレクトリの `index.md` として配置します。

| 種別           | 主仕様書の単位                  | 配置                                                       |
| -------------- | ------------------------------- | ---------------------------------------------------------- |
| Flow           | 独立した起動条件を持つ Flow     | `docs/specifications/flows/<flow>/index.md`                |
| LWC            | 独立した機能を提供する Bundle   | `docs/specifications/lwc/<bundle>/index.md`                |
| Aura           | Aura Component Bundle           | `docs/specifications/aura/<bundle>/index.md`               |
| Visualforce    | Visualforce Page                | `docs/specifications/visualforce/<page>/index.md`          |
| Apex Trigger   | Trigger ファイル                | `docs/specifications/apex/triggers/<trigger>/index.md`     |
| Apex Batch     | `Database.Batchable` 実装クラス | `docs/specifications/apex/batches/<batch>/index.md`        |
| Apex Scheduler | `Schedulable` 実装クラス        | `docs/specifications/apex/schedulers/<scheduler>/index.md` |
| Apex 非同期    | 独立した非同期処理              | `docs/specifications/apex/async/<process>/index.md`        |
| API / 連携     | 独立した外部契約または連携処理  | `docs/specifications/integrations/<integration>/index.md`  |
| イベント処理   | 独立した発行・購読機能          | `docs/specifications/events/<event>/index.md`              |

- Apex Trigger は一オブジェクト一 Trigger を原則とし、Trigger ファイルごとに主仕様書を作成する。
- 利用者、スケジュール、イベント、データ変更などから独立して起動する Flow は主仕様書を作成する。
- 一つの機能からだけ呼び出す Subflow は独立した主仕様書を作らず、呼び出し元の「対象実装・メタデータ」に完全な Flow API 名と役割を記載する。
- 複数機能から呼び出す共通 Subflow は主となる仕様を一つ決め、他の主仕様書から関連コンポーネントとして参照する。技術的に共通化されていることだけを理由に、利用者から独立した機能として扱わない。
- Invocable Apex は、原則として呼び出す Flow または自動化機能の「対象実装・メタデータ」に完全なクラス名と役割を記載する。複数機能から利用される場合も、主となる仕様を一つ決め、他の仕様から関連コンポーネントとして参照する。
- LWC Bundle は、独立した機能として主仕様書を作成するか、内部共通モジュールとして利用する主仕様書の「対象実装・メタデータ」に記載する。どちらにも分類されない Bundle を残さない。
- LWC、Aura、Visualforce では、画面を支える Apex Controller、Service、Selector なども画面側の主仕様書に含める。
- Batch と Scheduler は専用の組み合わせでも別々の主仕様書を作成し、相互にリンクする。
- 複数機能から利用される共通実装は、主となる仕様を一つ決め、他の仕様から関連コンポーネントとして参照する。
- 未実装の種別では空ディレクトリを作らず、`docs/specifications/index.md` に対象がないことを記載する。

### 実装からの逆引き

- 主仕様書の「対象実装・メタデータ」には、その機能を構成する入口メタデータ、本番 Apex クラス、LWC JavaScript ファイルを完全な名前で記載する。
- クラス名やモジュール名をワイルドカードで省略せず、実装ファイル名からリポジトリ検索で主仕様書へ到達できる状態にする。
- Apex テストクラス、Jest テストファイル、テストデータ生成クラスは「対象実装・メタデータ」の一覧に含めない。
- 仕様を直接検証する代表的なテストは「テスト・確認観点」でテストクラスまたはテストファイルとの対応を示す。すべての補助テストを列挙する必要はない。
- 対象が多い場合は、「対象実装・メタデータ」の中を入口、LWC、Apex、設定・権限などの小見出しで分ける。
- 「関連コンポーネント」には、別機能、Salesforce 標準機能、外部依存、詳細仕様へのリンクを記載し、「対象実装・メタデータ」の実装一覧と役割を混在させない。

## 主仕様書と詳細仕様

主仕様書には機能入口の全体像、処理一覧、実行順序、関連する詳細仕様へのリンクを記載します。詳細仕様には、利用者から見て独立したユースケースの具体的な振る舞いを記載します。

- Trigger では、Trigger から実行するユースケースごとに、処理が 1 つの段階から詳細仕様を作成する。
- LWC、Aura、Visualforce では、検索、登録、承認など、入力、権限、エラー、テスト観点が独立する処理を詳細仕様にする。
- Flow、Batch、Scheduler では、主仕様書だけで追跡しにくい独立処理がある場合に詳細仕様を作成する。
- クラス、Controller、Service、Selector などの技術レイヤーだけを理由に詳細仕様を分割しない。
- 主仕様書には詳細な判定条件や個別テストを重複させず、処理一覧から詳細仕様へ案内する。
- 詳細仕様には機能入口全体の説明を重複させず、対象ユースケースの条件と振る舞いだけを記載する。

例えば、取引先トリガーは次の構成にします。

```text
docs/specifications/apex/triggers/account-trigger/
├── index.md
├── account-name-normalization.md
├── account-owner-assignment.md
└── account-status-management.md
```

## ファイル名

- 機能入口のディレクトリ名は、API 名またはクラス名を内容の分かる kebab-case にする。
- 詳細仕様のファイル名は、クラス名ではなくユースケースを表す kebab-case にする。
- 主仕様書のファイル名は必ず `index.md` にする。
- 日付、連番、作業者名をファイル名に含めない。

## 仕様書索引

- `docs/specifications/index.md` の仕様書一覧は、Flow、LWC、Aura、Visualforce、Apex、API / 連携、イベント処理などの実装種別で分ける。
- Apex は対象が複数種別にまたがる場合、Trigger、Batch、Scheduler、非同期処理などの小見出しで分ける。
- 対象が存在するカテゴリだけ一覧の見出しを作り、対象がないカテゴリは一覧の末尾にまとめて記載する。
- 業務領域による分類は、承認済みの分類体系を確認できる場合だけ使用し、実装名から推測して作成しない。

## 主仕様書の構成

主仕様書は次の見出しをこの順序で使用します。

```markdown
# 機能名

## 概要

## 目的・利用場面

## 対象実装・メタデータ

## 入力

## 処理内容

## 出力・更新対象

## 権限・実行条件

## エラー処理

## 関連コンポーネント

## テスト・確認観点

## 制約・注意事項

## 既知の差異・確認事項
```

## 詳細仕様の構成

詳細仕様は次の見出しを必ずこの順序で使用します。該当する内容がない場合も見出しを残し、対象外であることと理由を記載します。

```markdown
# ユースケース名

## 概要

## 入力

## 処理内容

## 出力・更新対象

## 権限・実行条件

## エラー処理

## テスト・確認観点

## 制約・注意事項

## 既知の差異・確認事項
```

既存仕様書は、次に仕様影響がある変更を行うとき、または全体棚卸しするときにこの構成へ移行します。移行前の仕様書に「既知の差異・確認事項」がないことを、差異がないという判断には使用しません。

### 既知の差異・確認事項の書き方

要求や外部契約との比較状態を、次のいずれかで明記します。

| 状態         | 使用条件                                                             |
| ------------ | -------------------------------------------------------------------- |
| 差異あり     | 管理元を確認し、現行実装との不一致がある                             |
| 差異なし     | 管理元と比較対象を特定して確認し、現行実装との不一致がない           |
| 比較対象なし | 管理対象となる要求や外部契約がないことを確認できる                   |
| 未確認       | 管理元が不明、参照できない、または実装だけでは業務条件を判断できない |

差異ありまたは未確認の場合は、確認できた事実、影響、確認先を記載します。作業中だけ必要な調査メモや判断過程は置きません。

## 条件付き記載事項

次の事項は、該当する機能で主仕様書または詳細仕様に小見出しを追加して記載します。該当しない機能へ空の見出しを追加する必要はありません。

| 条件                         | 記載事項                                                                |
| ---------------------------- | ----------------------------------------------------------------------- |
| 大量データを処理する         | 想定件数、Governor Limits、分割単位、ロック、性能上の制約               |
| 非同期または定期実行する     | 起動条件、実行順序、重複実行、再試行、冪等性、監視、復旧                |
| 外部システムと連携する       | システム境界、契約、認証、タイムアウト、再試行、重複排除、障害時の扱い  |
| 重要データを保持・更新する   | データ所有者、整合性、保持・削除、監査、個人情報または機密情報の扱い    |
| 権限によって振る舞いが変わる | CRUD、FLS、共有、実行モード、外部・ゲストユーザー、権限不足時の期待動作 |
| 状態を遷移させる             | 状態、遷移条件、禁止遷移、並行更新時の扱い                              |
| 移行または互換性が関係する   | 既存データ、後方互換性、移行条件、ロールバック条件                      |

## 追跡可能性と責任分界

- 主仕様書から、対象実装、代表テスト、関連する権限、依存機能へ到達できるようにする。
- 恒久的な要求や外部契約に識別子または管理先がある場合は関連コンポーネントから参照する。個々の作業 PR、コミット、リリース履歴は GitHub を正とし、仕様書へ網羅的に転記しない。
- AI エージェントは、実装から確認できる現行実装仕様を更新できる。承認済み要求、業務判断、外部契約、セキュリティ判断を推測で作成または変更しない。
- 要求と実装の差異、セキュリティまたは個人情報への影響、外部契約の変更は、人間の確認事項として残す。

## 通常変更時の確認

振る舞いを変更するたびに仕様影響を判定します。一括更新はこの判定を代替せず、定期監査または明示依頼による再同期として扱います。

1. 関連する主仕様書と詳細仕様を特定する。
2. 入力、処理、出力、権限、エラー、制約、外部契約、運用方法への影響を確認する。
3. いずれかが変わる場合は仕様影響ありとし、原則として実装と同じ変更単位で仕様書を更新する。
4. リファクタリング、コメント、整形など、振る舞いが変わらない場合は更新不要とする。
5. 大規模移行などで仕様書更新を分離する場合は、差異、更新範囲、追跡先を変更報告へ記載する。

仕様書を追加、移動、削除した場合は、`docs/specifications/index.md` と関連リンクを更新します。

## 全体棚卸し

ユーザーから機能仕様書の更新、棚卸し、実装との再同期を依頼された場合は、AI エージェントが次の順序で対象全体を確認します。

1. `force-app/main/default` から、独自実装した Apex、Flow、LWC、Aura、Visualforce、API、連携、イベント処理の機能入口を棚卸しする。
2. `docs/specifications/index.md` と各主仕様書、詳細仕様の対応を確認する。
3. 実装、関連 Apex、メタデータ、権限判定、エラー処理、テストを読み、現在の振る舞いを確認する。
4. 新しい機能入口とユースケースを追加し、削除済みの機能に対応する仕様書と索引を削除する。
5. 実装を基準に、古い現行実装の説明、不足するユースケース、不要になった仕様書を更新する。
6. 承認済み要求や外部契約との比較状態を確認し、「既知の差異・確認事項」へ記録する。
7. 条件付き記載事項、代表テスト、権限、依存関係の不足を確認する。
8. 実装から確認できない内容は推測で補わず、確認できない範囲と確認先を明記する。
9. クラス名、ファイル名、索引、関連リンクを実装と目視で突き合わせる。
10. 「検証」のコマンドを実行する。
11. 更新内容、確認した実装、既知の差異、検証結果を GitHub Issue と Pull Request に記録する。

## 検証

- `npm run docs:check` で、一般的なリンク、見出し、ファイル名、索引からの到達性を確認する。
- `npm run prettier:verify` で整形を確認する。

機能仕様書と実装の自動同期や、仕様書専用チェッカーは運用の前提にしません。`npm run docs:check` は、必須見出し、対象機能の網羅性、仕様の内容、実装との一致を判定しません。変更時の仕様影響判定と定期的な全体棚卸しを組み合わせて最新性を維持します。
